# Styling & Dark Mode

---

## Dark Mode Flickering on Page Load

### The page briefly shows light mode before switching to dark

**Cause:** The theme is stored in `localStorage` and applied by a Zustand store after the React app hydrates. This creates a flash of the default (light) theme.

**Fix:** Add an inline `<script>` block to `index.html` that applies the theme class **before** React renders. Place it inside `<head>`:

```html
<!-- index.html -->
<head>
  <script>
    try {
      const theme = localStorage.getItem('theme') || 'system';
      const prefersDark = window.matchMedia('(prefers-color-scheme: dark)').matches;
      const isDark = theme === 'dark' || (theme === 'system' && prefersDark);
      if (isDark) document.documentElement.classList.add('dark');
    } catch (_) {}
  </script>
</head>
```

This script runs synchronously before the browser paints, eliminating the flash.

## CSS Variables Not Resolving

### Colors appear as `oklch(...)` text or render incorrectly in the browser

**Cause:** Your browser does not fully support `oklch()` color functions. Chrome 111+, Firefox 113+, and Safari 16.4+ are required.

**Fix:** Check your browser version. If you must support older browsers, add a fallback using `color-mix()` or traditional `hsl()` values alongside the `oklch()` declarations in `src/index.css`.

---

### A custom CSS class I added is not working

**Cause:** TailwindCSS v4 uses a **CSS-first** configuration model. Arbitrary values and custom utilities must be defined using `@theme` or `@utility` directives in `src/index.css`, not in a `tailwind.config.js` file.

**Fix:** Add custom tokens to the `@theme` block in `src/index.css`:

```css
@theme {
  --color-my-brand: oklch(60% 0.2 250);
  --font-my-font: 'Inter', sans-serif;
}
```

Then use them with standard Tailwind utility classes: `bg-my-brand`, `font-my-font`.

## Code Block Styling Issues

### Dark mode code block background is not transparent

**Cause:** Shiki injects inline `background-color` styles on the `.shiki` element which override CSS class-based rules.

**Fix:** The global CSS in `src/index.css` already uses `!important` to override this:

```css
.dark .shiki {
  background-color: transparent !important;
}
.dark .shiki span {
  background-color: transparent !important;
}
```

If you are still seeing a background, check whether a parent element has a `background-color` that is making the transparent background look odd, not Shiki itself.

---

### Code block syntax colors look wrong / all one color

**Cause:** The language identifier in your fenced code block is incorrect or missing, so Shiki falls back to plain text mode.

**Fix:** Use the correct language identifier:

```markdown
` `` `tsx       ✅ TypeScript + JSX
` `` `ts        ✅ TypeScript
` `` `js        ✅ JavaScript
` `` `bash      ✅ Shell / Bash
` `` `yaml      ✅ YAML
` `` `json      ✅ JSON
` `` `css       ✅ CSS
` `` `text      ✅ Plain text (no highlighting)
```

Check the [Shiki supported languages list](https://shiki.style/languages) for a complete reference.

## KaTeX Math Rendering Issues

### Math equations render as raw LaTeX text (e.g., `$x^2$`)

**Cause:** `remark-math` is not in the plugin list, or it is placed after `rehype-katex` (it must come first, in the remark phase).

**Fix:** Verify the plugin order in `vite.config.ts`:

```ts
{
  remarkPlugins: [
    remarkFrontmatter,
    remarkMdxFrontmatter,
    remarkGfm,
    remarkMath,      // ← Must be in remarkPlugins
  ],
  rehypePlugins: [
    rehypeSlug,
    rehypeKatex,     // ← Must be in rehypePlugins, after remarkMath runs
    rehypeAutolinkHeadings,
    rehypeShiki,
  ],
}
```

---

### KaTeX renders but looks unstyled (raw HTML, no font)

**Cause:** The KaTeX CSS stylesheet is not being imported.

**Fix:** Ensure this line is present in `src/index.css`:

```css
@import "katex/dist/katex.min.css";
```

---

### Block math overflows horizontally on mobile

**Cause:** Complex equations are wider than the viewport on small screens.

**Fix:** This is handled globally with a CSS rule in `src/index.css`. If you see overflow, verify the rule exists:

```css
.katex-display {
  overflow-x: auto;
  overflow-y: hidden;
  padding: 1rem 0;
}
```

## Docs Sidebar Issues

### Sidebar shows "No docs found" even though files exist

**Cause:** One of the following:
1. The MDX files are not inside `/content/docs/` (glob pattern mismatch).
2. All docs have `draft: true` in frontmatter.
3. The `section` field is missing from the frontmatter.

**Fix checklist:**
```
✅ File path: content/docs/<section>/<page>.mdx
✅ frontmatter has section: "My Section"
✅ frontmatter does NOT have draft: true
✅ npm run dev was restarted after adding the file
```

---

### Sidebar section order is wrong

**Cause:** The sidebar sorts sections in the order they are first encountered in the loaded docs. The `order` field controls items **within** a section, not the section itself.

**Fix:** Set a low `order` number on the first page of each section you want to appear earlier. Since sections are keyed by `sectionSlug` (the folder name), name your folders alphabetically if you need predictable ordering:

```
content/docs/01-introduction/   → appears first
content/docs/02-guides/         → appears second
content/docs/03-troubleshooting/ → appears third
```

Alternatively, set unique `order` values that span sections:

```yaml
# introduction/index.mdx
order: 1

# guides/installation.mdx
order: 10

# troubleshooting/index.mdx
order: 20
```

<Callout type="note">
Section grouping in the sidebar is determined by the `sectionSlug` (folder name), not the `section` string in frontmatter. The `section` string is only used as the human-readable label displayed in the sidebar.
</Callout>
