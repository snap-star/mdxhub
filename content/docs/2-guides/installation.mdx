Follow these steps to get ReactMDX running on your local machine.

## Prerequisites

Before you begin, make sure you have the following installed:

- **Node.js 20+** — [Download here](https://nodejs.org)
- **npm 10+** or **pnpm 9+**

## Setup

### 1. Clone the repository

```bash
git clone https://github.com/your-username/reactmdx.git
cd reactmdx
```

### 2. Install dependencies

Use `npm` or `pnpm` depending on your preference:

```bash
npm install
# or
pnpm install
```

### 3. Start the development server

```bash
npm run dev
# or
pnpm run dev
```

Your site is now running at `http://localhost:3000`. Vite will automatically hot-reload whenever you change source files **or** content files.

<Callout type="tip">
Any `.md` or `.mdx` file you add to `content/blog/` or `content/docs/` will be automatically discovered — no registration or routing config needed!
</Callout>

## Content Discovery

This project discovers content with Vite's `import.meta.glob` patterns:

- `content/blog/**/*.{md,mdx}` for blog posts
- `content/docs/**/*.{md,mdx}` for documentation pages

Nested folders are supported, and `index.mdx` inside a folder maps to the parent route slug.

### Example slug mappings

```text
content/blog/my-post.mdx                  → /blog/my-post
content/blog/tutorials/react-hooks.mdx     → /blog/tutorials/react-hooks
content/blog/tutorials/react/index.mdx     → /blog/tutorials/react
content/docs/guides/installation.mdx       → /docs/guides/installation
content/docs/reference/index.mdx           → /docs/reference
content/docs/reference/api/usage.mdx       → /docs/reference/api/usage
content/about.mdx                           → /about
```

### Valid vs invalid docs/about slugs

Valid paths:
- `/about` → `content/about.mdx`
- `/docs` → docs index page handled by the docs layout
- `/docs/guides/installation` → `content/docs/guides/installation.mdx`
- `/docs/reference/api` → `content/docs/reference/api/index.mdx`
- `/docs/reference/api/usage` → `content/docs/reference/api/usage.mdx`

Avoid these invalid paths:
- `/about/me` — only `content/about.mdx` maps to `/about`
- `/docs/about` — docs content must be inside `content/docs/`, not `content/about.mdx`
- `/content/docs/guides/installation` — URLs do not include the `content/` prefix
- `/blog/installation` for a file stored in `content/docs/` — blog and docs are separate route trees

## Static Assets

- Place images, icons, or any static file in the `public/` folder.
- Use `/path/to/file.png` in your MDX or frontmatter to reference it.

### Relative content assets

MDX image sources that use relative paths are resolved relative to the current content folder. For example:

```mdx
![Diagram](./diagram.png)
```

If the MDX file is at `content/blog/tutorials/react/index.mdx`, the image is resolved from `content/blog/tutorials/react/diagram.png`.

## Project Configuration

### Site Config

Open `site.config.json` at the project root to configure the GitHub link that appears in the navbar badge:

```json
{
  "githubUrl": "https://github.com/your-username/your-repo"
}
```

The version badge next to the GitHub icon is automatically pulled from your `package.json` `version` field.

### Authors Registry

Register content authors by editing `content/authors/authors.yaml`:

```yaml
- id: your-id             # Used in blog frontmatter
  name: "Your Full Name"
  avatar: "/your-avatar.png"   # Can be a local /public path or a full URL
  bio: "Short bio."
  twitter: yourhandle
  github: yourhandle
  website: https://yoursite.com
```

### Adding Your Profile Photo

Place your avatar image inside the `public/` directory. It will be served at the root path. For example, `public/snap-star.png` is accessible as `/snap-star.png`.

## Building for Production

When you're ready to deploy:

```bash
npm run build
# or
pnpm run build
```

This runs the TypeScript compiler (`tsc`) and then Vite's production bundler. All MDX content is compiled at build time into optimized JavaScript. The output is placed in the `dist/` directory and is fully static — no server required.

```bash
npm run preview
# or
pnpm run preview
```

<Callout type="warning">
Because this is a Single Page Application (SPA), certain hosting providers need special configuration to handle client-side routing. See the [Deployment Guide](/docs/deployment/hosting) for Vercel, Netlify, and GitHub Pages instructions.
</Callout>

## Key Scripts

| Command | Description |
| :--- | :--- |
| `npm run dev` | Start the Vite dev server with HMR (also generates content-index.json, RSS, and sitemap) |
| `npm run build` | Type-check and build for production |
| `npm run preview` | Serve the `dist/` output locally |
| `npm run lint` | Run ESLint on all source files |
| `npm run dev:watch` | Watch `content/` for changes and auto-regenerate content-index.json, RSS, and sitemap |

### Dev Watch Script

For a smoother development experience, run `npm run dev:watch` in a **separate terminal** alongside `npm run dev`:

```bash
# Terminal 1 — Vite dev server with HMR
npm run dev

# Terminal 2 — content file watcher (auto-regenerates content-index.json on saves)
npm run dev:watch
```

The watcher monitors all `.md`, `.mdx`, `.yaml`, and `.yml` files under `content/`. Any time you save a file, the script:

1. Regenerates `public/content-index.json` (with tag parsing, featured/cc/series fields, etc.)
2. Regenerates `public/rss.xml`
3. Regenerates `public/sitemap.xml`

This means the Featured Posts widget, Series navigation, category/tag filters, and CC license badges all reflect your latest edits instantly — no full rebuild required.
