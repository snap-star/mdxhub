# 04 — Zustand v5

## Install

```bash
pnpm add zustand
```

- Current stable: v5.0.x. ~1.2KB, no providers required, hook-based API.

## Why Zustand (vs Context / Redux)

- No `<Provider>` wrapping needed — store is just a hook.
- Selective subscriptions: components only re-render when the slice they read changes.
- Minimal boilerplate vs Redux Toolkit; better for frequently-changing state than Context.

## Basic store

```ts
import { create } from "zustand";

interface BearState {
  bears: number;
  increase: () => void;
  reset: () => void;
}

export const useBearStore = create<BearState>((set) => ({
  bears: 0,
  increase: () => set((state) => ({ bears: state.bears + 1 })),
  reset: () => set({ bears: 0 }),
}));
```

```tsx
function BearCounter() {
  const bears = useBearStore((state) => state.bears); // selector → only re-renders on `bears` change
  return <h1>{bears}</h1>;
}

function Controls() {
  const increase = useBearStore((state) => state.increase);
  return <button onClick={increase}>+1</button>;
}
```

⚠️ Avoid `useBearStore()` with no selector — re-renders on **every** state change.

## Async actions

```ts
const useFishStore = create<{ fish: Fish[]; fetchFish: () => Promise<void> }>((set) => ({
  fish: [],
  fetchFish: async () => {
    const res = await fetch("/api/fish");
    set({ fish: await res.json() });
  },
}));
```

## Accessing state outside React

```ts
const useSoundStore = create<{ sound: string; play: () => void }>((set, get) => ({
  sound: "grunt",
  play: () => playAudio(get().sound), // get() reads current state non-reactively
}));

// Outside components:
useSoundStore.getState().sound;
useSoundStore.setState({ sound: "roar" });
```

## Middleware

```ts
import { create } from "zustand";
import { persist, devtools, subscribeWithSelector } from "zustand/middleware";
import { immer } from "zustand/middleware/immer";

const useStore = create<State>()(
  devtools(
    persist(
      immer((set) => ({
        todos: [],
        addTodo: (text: string) =>
          set((state) => {
            state.todos.push({ id: crypto.randomUUID(), text }); // mutate-style with immer
          }),
      })),
      { name: "todo-storage" }, // localStorage key
    ),
  ),
);
```

| Middleware | Purpose |
|---|---|
| `persist` | Sync store to localStorage/sessionStorage |
| `devtools` | Redux DevTools integration |
| `immer` | Write "mutating" updates safely |
| `subscribeWithSelector` | Subscribe to specific slices outside React |

## Slices pattern (large apps)

```ts
const createUserSlice = (set: SetState) => ({
  user: null,
  setUser: (user: User) => set({ user }),
});

const createCartSlice = (set: SetState) => ({
  cart: [],
  addToCart: (item: Item) => set((s) => ({ cart: [...s.cart, item] })),
});

export const useAppStore = create<UserSlice & CartSlice>()((...a) => ({
  ...createUserSlice(...a),
  ...createCartSlice(...a),
}));
```

- Keeps one store but organizes logic into independent, composable slices.

## Where Zustand fits vs tRPC/React Query

| State type | Tool |
|---|---|
| Server data (posts, users from DB) | tRPC + React Query cache — **not** Zustand |
| UI/client-only state (modals, theme, form drafts, cart before checkout) | Zustand |
| Derived/computed values | Selector functions inside the store |

Don't duplicate server data into Zustand — let React Query own server cache; use Zustand only for genuinely client-side state.
