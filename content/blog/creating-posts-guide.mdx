## How Content Discovery Works

This project uses **Vite's `import.meta.glob`** to automatically discover and load `.md` and `.mdx` content at build time. You do not register files manually — simply drop them into the right folder and they appear.

```
content/
├── blog/             ← All blog posts go here
│   ├── my-post.mdx
│   └── sub-folder/
│       └── nested-post.mdx   ← Nested files supported!
├── docs/             ← All documentation pages go here
│   └── guides/
│       └── getting-started.mdx
├── authors/
│   └── authors.yaml  ← Author registry
└── about.mdx         ← Special page (rendered at /about)
```

The discovery rules are:
- **Blog posts**: Any `.md` or `.mdx` file inside `/content/blog/**` is automatically a blog post.
- **Doc pages**: Any `.md` or `.mdx` file inside `/content/docs/**` is automatically a documentation page.
- **Slug derivation**: `content/blog/my-post.mdx` → `/blog/my-post`. A file named `index.mdx` inside a folder maps to the folder slug.
- **Draft posts**: Set `draft: true` in frontmatter to exclude a post from the published feed.
- **Sorting**: Blog posts use explicit `order` first, then date descending. Doc pages are sorted by `order` ascending.

## Part 1: Creating a Blog Post

### Step 1 — Create the File

Create a new `.md` or `.mdx` file inside `/content/blog/`. The file path becomes the page slug.

```
content/blog/my-first-post.mdx             → /blog/my-first-post
content/blog/tutorials/react/index.mdx      → /blog/tutorials/react
content/blog/series/part-1.mdx              → /blog/series/part-1
```

### Step 2 — Write the YAML Frontmatter

Every blog post **must** begin with a YAML frontmatter block delimited by `---`. Here is the full schema:

```yaml
---
title: "My Awesome Post"         # Required — displayed as the page title and in cards
date: "2026-06-21"               # Required — ISO 8601 date (YYYY-MM-DD), used for sorting
author: "chigusa-asuha"          # Required — must match an id in content/authors/authors.yaml
category: "Tutorial"             # Required — used for category filtering
tags: ["react", "mdx", "vite"]  # Required — array of strings for tag filtering and search
description: "A short summary."  # Optional but recommended — shown on cards and SEO meta tags
coverImage: "https://..."        # Optional — thumbnail shown on the blog homepage card
featured: true                    # Optional — highlights this post in the Featured widget on the homepage
cc: "CC-BY-4.0"                  # Optional — shows a Creative Commons license badge
readingTime: 5                    # Optional — override auto-calculated reading time (minutes)
draft: true                       # Optional — set to true to hide from the published feed
series: "My Series"             # Optional — group related posts into a navigable series
seriesOrder: 1                    # Optional — used to sort posts within a series
---
```

<Callout type="warning" title="Required Fields">
If you omit `title`, `date`, `author`, `category`, or `tags`, the post may not render correctly. All five are required.
</Callout>

<Callout type="tip" title="Cover Images">
Set `coverImage` to a URL or a static `/public/` path. It appears as the blog card thumbnail and as the hero image inside the post.
</Callout>

<Callout type="info" title="Series Grouping">
Use `series` and `seriesOrder` together to group related posts. Posts sharing the same `series` name will appear in a **Series Navigation** widget at the top and bottom of each post, with prev/next links, a progress bar, and a scrollable list. Set `seriesOrder` to control the order (e.g., 1, 2, 3).
</Callout>

### Step 3 — Start Writing

Below the closing `---`, write your content using standard Markdown. All the supported syntax is detailed below.

## Part 2: Creating a Docs Page

Documentation pages live in `/content/docs/` and are organized into **sections** using subdirectories. Nested folders are fully supported.

### File Structure

```
content/docs/
├── introduction/
│   └── index.mdx           → /docs/introduction
├── guides/
│   ├── installation.mdx    → /docs/guides/installation
│   └── advanced/
│       └── features.mdx    → /docs/guides/advanced/features
├── reference/
│   └── api/
│       └── usage.mdx       → /docs/reference/api/usage
└── troubleshooting/
    └── index.mdx           → /docs/troubleshooting
```

### Docs Frontmatter Schema

```yaml
---
title: "Getting Started"        # Required — page title shown in sidebar and heading
section: "Introduction"         # Required — the human-readable section name
order: 1                         # Optional — controls sidebar sort order within a section
description: "Learn how to…"   # Optional — shown in search results and browser tab
draft: false                    # Optional — hides page from nav and search when true
version: "v1.0"                # Optional — displays a version badge
toc: true                        # Optional — enables the table of contents (default: true)
prev: "introduction"            # Optional — slug of the previous doc
next: "guides/advanced/features" # Optional — slug of the next doc
---
```

### Sidebar Section Order

The sidebar sections are sorted by folder order and the `section` frontmatter label. Use `order` to control page order within a section.

## Part 3: Managing Authors

Authors are stored in a single YAML registry at `/content/authors/authors.yaml`. Add a new entry to register an author:

```yaml
- id: chigusa-asuha            # Must be unique — used in post frontmatter
  name: "Chigusa Asuha"
  avatar: "/snap-star.png"     # Can be a local /public path or a full URL
  bio: "Lead Developer & Creative Technologist."
  twitter: chigusa
  github: chigusa-asuha
  website: https://example.com
```

After adding the entry, reference the author in any blog post's frontmatter using the `id`:

```yaml
author: "chigusa-asuha"
```

## Part 4: Full MDX Syntax Reference

### Standard Markdown

All standard Markdown is supported:

```markdown
# H1 Heading
## H2 Heading
### H3 Heading

**Bold text**, _italic text_, ~~strikethrough~~

[Link text](https://example.com)
[Internal link](/blog/another-post)

> A blockquote stands out with a branded left border.

- Unordered list item
- Another item
  - Nested item

1. Ordered list
2. Second item

---
A horizontal rule
```

### Code Blocks

Code blocks use **Shiki** with Monokai in dark mode and GitHub Light in light mode. Specify the language after the opening triple backtick for syntax highlighting:

````markdown
```javascript
const greeting = "Hello, world!";
console.log(greeting);
```

```tsx
export function Button({ children }: { children: React.ReactNode }) {
  return <button className="btn">{children}</button>;
}
```

```bash
npm install && npm run dev
```

```yaml
name: Deploy
on:
  push:
    branches: [main]
```
````

The language label is automatically displayed in the top-right corner of every code block.

### Tables

GFM-style tables are fully supported with responsive horizontal scrolling:

```markdown
| Column A | Column B | Column C |
| :------- | :------: | -------: |
| Left     | Center   | Right    |
| Data     | Data     | Data     |
```

| Alignment | Syntax | Effect |
| :-------- | :----: | -----: |
| Left | `:---` | Default |
| Center | `:---:` | Centered |
| Right | `---:` | Right-aligned |

### Mathematical Equations (KaTeX)

Full LaTeX math is supported via `remark-math` and `rehype-katex`.

**Inline math** — wrap with single `$`:

```markdown
The quadratic formula is $x = \frac{-b \pm \sqrt{b^2 - 4ac}}{2a}$.
```

**Block math** — wrap with double `$$`:

```markdown
$$
\nabla \cdot \mathbf{E} = \frac{\rho}{\varepsilon_0}
$$
```

## Part 5: React Components in MDX

All components below are **globally available** in every MDX file — no import needed.

### Callout / Admonition

```mdx
<Callout type="note" title="Custom Title">
  This is an informational callout. The title is optional.
</Callout>

<Callout type="tip">A helpful tip for the reader.</Callout>

<Callout type="warning">Something the reader should be careful about.</Callout>

<Callout type="danger">A critical error or dangerous action warning.</Callout>
```

**Live examples:**

<Callout type="note" title="Note (Blue)">
  Used for supplementary information, context, or background details.
</Callout>

<Callout type="tip" title="Tip (Green)">
  Used for helpful suggestions, best practices, or time-saving tricks.
</Callout>

<Callout type="warning" title="Warning (Yellow)">
  Used for non-destructive cautions — things that may cause unexpected behavior.
</Callout>

<Callout type="danger" title="Danger (Red)">
  Used for critical warnings — data loss risks, security issues, or breaking changes.
</Callout>

### VideoEmbed

Embed YouTube videos, Shorts, Vimeo, or any iframe-compatible URL directly in your content:

```mdx
{/* Full YouTube URL */}
<VideoEmbed src="https://www.youtube.com/watch?v=VIDEO_ID" />

{/* Short youtu.be link */}
<VideoEmbed src="https://youtu.be/VIDEO_ID" caption="Optional caption below" />

{/* Custom aspect ratio */}
<VideoEmbed src="https://youtu.be/VIDEO_ID" aspect="4/3" />

{/* Vertical Shorts */}
<VideoEmbed src="https://www.youtube.com/shorts/VIDEO_ID" aspect="9/16" />
```

**Supported `aspect` values:** `16/9` (default), `4/3`, `1/1`, `9/16`

All formats are auto-detected and resolved to proper embed URLs. Videos are lazy-loaded for performance.

### ProfileBadge

Renders your author profile card (used on the About page):

### Creative Commons License

Add a license badge at the end of a post using the `cc` frontmatter key, or inline:

```mdx
<CCLicense code="CC-BY-SA-4.0" author="Chigusa Asuha" />
```

**Supported `cc` codes:**

| Code | License Name |
| :--- | :--- |
| `CC-BY-4.0` | Creative Commons Attribution 4.0 |
| `CC-BY-SA-4.0` | Creative Commons Attribution-ShareAlike 4.0 |
| `CC-BY-NC-4.0` | Creative Commons Attribution-NonCommercial 4.0 |
| `CC0-1.0` | Creative Commons Zero (Public Domain) |

### Badge

The `<Badge>` component renders a small, colorful pill perfect for tags, labels, status indicators, and metadata. Supports 22+ Tailwind color names, semantic admonition variants, and 40+ built-in icons.

```mdx
{/* Semantic variants */}
<Badge variant="tip">Tip</Badge>
<Badge variant="warning">Caution</Badge>
<Badge variant="danger">Danger</Badge>

{/* Direct Tailwind colors */}
<Badge variant="emerald">Emerald</Badge>
<Badge variant="violet">Violet</Badge>
<Badge variant="rose">Rose</Badge>

{/* With icons */}
<Badge variant="success" icon="check">Completed</Badge>
<Badge variant="warning" icon="zap">Requires Attention</Badge>
<Badge variant="info" icon="book">Documentation</Badge>

{/* Inline with text */}
Run <Badge variant="warning" icon="zap">npm install</Badge> before starting.
```

**Admonition-style variants:** `note` (slate), `tip` (emerald), `info` (blue), `success` (green), `warning` (amber), `danger` (red), `destructive` (red)

**Available icon names:** `check`, `copy`, `info`, `warning`, `danger`, `star`, `heart`, `zap`, `shield`, `clock`, `calendar`, `book`, `file`, `tag`, `hash`, `mail`, `link`, `sun`, `moon`, `cloud`, `download`, `upload`, `refresh`, `plus`, `minus`, `close`, `search`, `settings`, `user`, `home`, `arrow`, `external`, `github`, `globe`, `bulb`, `rocket`, `sparkles`, `target`, `flag`

### Tooltip

The `<Tooltip>` component shows a floating label when the user hovers or focuses the wrapped text. The trigger appears with a dotted underline to indicate interactivity.

```mdx
<Tooltip content="Server Components that run on the server">RSC</Tooltip>
enables efficient data fetching.

{/* Placement control */}
<Tooltip content="Appears below" side="bottom">Hover me</Tooltip>
<Tooltip content="Appears to the right" side="right">Me too</Tooltip>

{/* Combined with Badge */}
<Badge variant="blue" icon="shield">
  <Tooltip content="Application Programming Interface">API</Tooltip>
</Badge>
```

**Props:** `content` (string, required), `side` (`"top"` | `"bottom"` | `"left"` | `"right"`, default `"top"`)

### Image Lightbox

**No component needed.** Every image rendered through the standard `img` tag is automatically clickable and opens in a full-screen lightbox viewer with zoom, rotate, and close-on-Escape controls.

Simply use standard Markdown image syntax:

```mdx
![Descriptive alt text](./diagram.png)
```

Clicking the image opens a full-screen overlay with:
- Zoom in/out (25%–400%) and rotate (90° increments) via the bottom toolbar
- **Escape** key or clicking outside the image to dismiss
- **Alt text** displayed as a caption in the overlay

## Part 6: SEO

Every page automatically gets SEO meta tags, Open Graph, and Twitter Card tags injected via `react-helmet-async`.

For blog posts, the SEO data is auto-populated from frontmatter:
- **Title** → `title`
- **Description** → `description`
- **OG Image** → `coverImage`

No manual configuration is needed — it works automatically.

## Dev Workflow: Content Watch Script

For a smoother development experience, run `npm run dev:watch` in a **separate terminal** alongside `npm run dev`:

```bash
# Terminal 1 — Vite dev server
npm run dev

# Terminal 2 — auto-regenerates content-index.json, RSS, and sitemap on every save
npm run dev:watch
```

The watcher monitors all `.md`, `.mdx`, `.yaml`, and `.yml` files under `content/`. Any time you save a file, it automatically:

1. Regenerates `public/content-index.json` (posts, tags, featured, series, cc, etc.)
2. Regenerates `public/rss.xml`
3. Regenerates `public/sitemap.xml`

This means your **Featured Posts** widget, **Series navigation**, **category/tag filters**, and **CC license badges** all update instantly — no full rebuild needed.
